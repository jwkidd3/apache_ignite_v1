Understanding the Evolution of Apache Ignite

The Big Picture

A Complete Rewrite, Not an Upgrade

Apache Ignite 3.0 is a ground-up architectural rewrite - it's not just an incremental update from 2.x. Think of it as a new product built on lessons learned from Ignite 2.x.

Philosophy Shift

2.xCache-First

  • Data accessed via Cache API
  • Key-value operations primary
  • SQL layered on top

3.xSchema-First

  • Schema defined upfront
  • SQL is first-class citizen
  • Memory-centric storage

Architecture Comparison

Core Architectural Differences

Consensus and Replication

2.16

  • Custom replication protocol
  • Baseline topology for persistence
  • Manual partition management
  • Split-brain handling varies

3.x

  • RAFT consensus algorithm
  • Automatic leader election
  • Built-in split-brain protection
  • Zone-based replication (3.1+)

Storage Architecture

Ignite 2.16

  • On-heap + Off-heap memory regions
  • Native persistence (optional)
  • Single storage engine

Ignite 3.x

  • Pluggable storage engines:
    • aimem - Pure in-memory storage
    • aipersist - Memory-first with persistence
    • RocksDB - Disk-based for write-heavy workloads

Architecture Diagram

Ignite 2.x Architecture: ┌─────────────────────────────────────────────────────┐ │ Application │ ├─────────────────────────────────────────────────────┤ │ Cache API │ SQL Engine │ Compute Grid │ ├─────────────────────────────────────────────────────┤ │ Data Grid (Key-Value + Partitioning) │ ├─────────────────────────────────────────────────────┤ │ On-Heap │ Off-Heap │ Native Persistence │ └─────────────────────────────────────────────────────┘ Ignite 3.x Architecture: ┌─────────────────────────────────────────────────────┐ │ Application │ ├─────────────────────────────────────────────────────┤ │ Table API │ Key-Value API │ SQL API │ ├─────────────────────────────────────────────────────┤ │ Unified Schema Layer (RAFT-based) │ ├─────────────────────────────────────────────────────┤ │ aimem │ aipersist │ RocksDB (pluggable) │ └─────────────────────────────────────────────────────┘

API Changes

From Cache API to Table API

Ignite 2.16: Cache API

// Starting Ignite 2.x
Ignite ignite = Ignition.start();

// Creating a cache
IgniteCache<Integer, Person> cache = ignite.getOrCreateCache("persons");

// CRUD operations
cache.put(1, new Person("John", "Doe"));
Person person = cache.get(1);
cache.remove(1);

// SQL query
SqlFieldsQuery query = new SqlFieldsQuery(
    "SELECT * FROM Person WHERE lastName = ?");
query.setArgs("Doe");
List<List<?>> results = cache.query(query).getAll();

Ignite 3.x: Table API

// Starting Ignite 3.x
IgniteClient client = IgniteClient.builder()
    .addresses("localhost:10800")
    .build();

// Getting a table (schema-first approach)
Table<Tuple> table = client.tables().table("persons");

// CRUD operations using Tuples
Tuple key = Tuple.create().set("id", 1);
Tuple value = Tuple.create()
    .set("id", 1)
    .set("firstName", "John")
    .set("lastName", "Doe");

table.recordView().upsert(null, value);
Tuple person = table.recordView().get(null, key);

// SQL query
ResultSet<SqlRow> rs = client.sql()
    .execute(null, "SELECT * FROM persons WHERE lastName = ?", "Doe");

Key-Value View in 3.x

// Ignite 3.x also supports Key-Value views
KeyValueView<Integer, Person> kvView =
    table.keyValueView(Integer.class, Person.class);

// Similar to cache operations
kvView.put(null, 1, new Person("John", "Doe"));
Person person = kvView.get(null, 1);
kvView.remove(null, 1);

Note: Ignite 3.x provides multiple views of the same data - RecordView, KeyValueView, and SQL - all accessing the same underlying storage.

Client Model Changes

2.16

  • Thick Client: Full functionality, joins cluster
  • Thin Client: Lightweight, limited features
  • Must choose based on needs

3.x

  • Unified client model
  • All features via thin protocol
  • No thick/thin distinction
  • Consistent API across languages

SQL Improvements in 3.x

SQL as a First-Class Citizen

Transaction Support

Ignite 2.16

  • SQL transactions limited
  • Key-Value transactions more robust
  • Cannot mix SQL and KV in same transaction easily

Ignite 3.x

  • Full SQL transaction support
  • Strictly serializable isolation
  • Mix SQL and KV operations in same transaction
  • RDBMS migration-friendly

SQL Transaction Example (3.x)

// Ignite 3.x - Full transactional SQL
try (Transaction tx = client.transactions().begin()) {
    // SQL operations
    client.sql().execute(tx,
        "INSERT INTO accounts (id, balance) VALUES (?, ?)", 1, 1000);

    // Key-Value operations in same transaction
    KeyValueView<Integer, Account> kv = accountsTable.keyValueView();
    kv.put(tx, 2, new Account(2, 2000));

    // Both committed atomically
    tx.commit();
} catch (Exception e) {
    // Automatic rollback on exception
}

Calcite SQL Engine

  • 2.16 H2-based SQL engine (legacy) + Calcite (newer)
  • 3.x Apache Calcite only - modern, optimized

Benefits of Calcite in 3.x:

  • Better query optimization
  • More SQL standard compliance
  • Improved distributed join performance
  • Cost-based query planning

Schema Management

Defining Data Structures

Ignite 2.16: Annotation-Based Schema

// Define schema via annotations
public class Person implements Serializable {
    @QuerySqlField(index = true)
    private int id;

    @QuerySqlField
    private String firstName;

    @QuerySqlField(index = true)
    private String lastName;
}

// Register with cache configuration
CacheConfiguration<Integer, Person> cfg = new CacheConfiguration<>("persons");
cfg.setIndexedTypes(Integer.class, Person.class);
IgniteCache<Integer, Person> cache = ignite.getOrCreateCache(cfg);

Ignite 3.x: SQL DDL Schema

-- Create table using SQL DDL
CREATE TABLE persons (
    id INT PRIMARY KEY,
    firstName VARCHAR(100),
    lastName VARCHAR(100)
) WITH storage_profile='default';

-- Create indexes
CREATE INDEX idx_lastname ON persons (lastName);

-- Alter table schema
ALTER TABLE persons ADD COLUMN email VARCHAR(200);

-- All schema changes are dynamic and consistent across cluster

Key Change: Schema is defined via SQL DDL, not Java annotations. This makes it language-agnostic and familiar to database users.

Schema Evolution

Operation Ignite 2.16 Ignite 3.x
Add column Requires restart or workaround ALTER TABLE
Add index Possible but complex CREATE INDEX
Change column type Not supported Limited support
Drop column Not supported ALTER TABLE DROP

CLI and Tooling

Management and Operations

Ignite 2.16: Control Script

# Activate cluster
./control.sh --activate

# Check cluster state
./control.sh --state

# Set baseline topology
./control.sh --baseline

# Deactivate cluster
./control.sh --deactivate

# Various JMX-based operations

Ignite 3.x: Unified CLI

# Initialize cluster
ignite3 cluster init --name=myCluster --meta-storage-node=node1

# Check cluster status
ignite3 cluster status

# Run SQL directly from CLI
ignite3 sql "SELECT * FROM persons"

# Dynamic configuration
ignite3 cluster config show
ignite3 cluster config update compute.threadPoolSize=16

# Node management
ignite3 node status
ignite3 node config show

When to Use Each Version

Ignite 2.16 Strengths

  • Mature and battle-tested - Production-proven at scale
  • Feature complete - All features available today
  • Rich ecosystem - Spring, Hibernate, Kafka integrations
  • Compute Grid - Full MapReduce, closures, services
  • Near Cache - Client-side caching for hot data
  • Continuous Queries - Real-time data notifications

Ignite 3.x Strengths

  • Better consistency - RAFT provides stronger guarantees
  • SQL-first workflows - Better SQL transaction support
  • Simpler operations - Modern CLI, dynamic config
  • Flexible storage - Right engine for the workload
  • Future-proof - Active development focus
  • Clean architecture - No legacy baggage

Migration Considerations

From 2.x to 3.x

Important: No Direct Migration Path

There is no in-place upgrade from Ignite 2.x to 3.x. The architectural differences are too significant. Migration requires:

  1. Setting up a new Ignite 3.x cluster
  2. Recreating schema using SQL DDL
  3. Migrating data (export/import or dual-write)
  4. Updating application code

Code Migration Examples

// Ignite 2.x
IgniteCache<Integer, Person> cache = ignite.cache("persons");
cache.put(1, person);
Person p = cache.get(1);

// Ignite 3.x equivalent
Table personTable = client.tables().table("persons");
KeyValueView<Integer, Person> kv = personTable.keyValueView(Integer.class, Person.class);
kv.put(null, 1, person);  // null = no transaction
Person p = kv.get(null, 1);

Migration Toolkit (3.1+)

Apache Ignite 3.1 includes migration tools:

  • DDL Generator - Convert 2.x schema to 3.x SQL
  • Data Migration - Export from 2.x, import to 3.x
  • Progress Tracking - Monitor migration status
  • Type Conversion - Handle legacy Java types
# Example migration command
ignite3 migration generate-ddl --source-config=ignite2-config.xml
ignite3 migration export-data --source-config=ignite2-config.xml --output=data.bin
ignite3 migration import-data --input=data.bin

Migration Strategy

  1. Assess Current Usage
    • Which features do you use?
    • Are they available in 3.x?
  2. Plan Parallel Operation
    • Run both versions during transition
    • Dual-write if needed
  3. Migrate Incrementally
    • Start with new features on 3.x
    • Move existing workloads gradually
  4. Test Thoroughly
    • Behavior differences exist
    • Performance characteristics differ

Summary

Key Takeaways

Ignite 2.16 - Choose When

  • You need production-proven stability
  • You require Compute Grid features
  • You need Continuous Queries
  • You need Service Grid
  • You have existing 2.x deployment
  • You need ML/Streaming features

Ignite 3.x - Choose When

  • Starting a new project
  • SQL is your primary interface
  • You need strong consistency (RAFT)
  • You're migrating from RDBMS
  • You want modern tooling
  • You want pluggable storage

The Bottom Line

Ignite 2.16 is mature, feature-complete, and production-proven. It's the safe choice for existing workloads.

Ignite 3.x is the future, with a cleaner architecture and better SQL support. Choose it for new SQL-heavy projects.

Both versions will coexist - Ignite 2.x continues maintenance releases
